## Handling "Pre-existing" Failures
If you claim a failure is pre-existing:
1. `git stash` your changes
2. Run the failing test on main: `git checkout main && bun run test:e2e`
3. If it fails on main: Create a separate issue for it, then `git checkout -` and `git stash pop`
4. If it passes on main: The failure is YOUR regression. Fix it.
5. Document your investigation in the commit message

## Handling "Flaky" Tests
If you claim a test is flaky:
1. Run the test 10 times: `for i in {1..10}; do bun run test:e2e --grep "test name"; done`
2. If it fails inconsistently: Fix the flakiness (better waits, deterministic data)
3. If it passes consistently: Your code broke it. Fix your code.
4. NEVER skip a flaky test without fixing it

## Handling "Unrelated" Failures
If you claim a failure is unrelated to your changes:
1. `git diff main...HEAD` — what files did you change?
2. Does the failing test touch any of those files/features?
3. If yes: It's related. Fix it.
4. If no: Document why it's unrelated, create a separate issue, proceed cautiously.
